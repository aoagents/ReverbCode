Electron vs Tauri: picking the desktop shell

The desktop app is a thin control plane over an external Go daemon, with multiple live xterm.js terminal panes. This is the decision record: what was weighed, how the answer moved as constraints firmed up and evidence accumulated, and where it finally landed.

Electron shell · macOS + Windows · React renderer · thin TypeScript supervisor over the Go daemon

The deciding axis: language count

Once Linux is gone, the other axes flatten (see scorecard). The one with real, durable weight is how many languages the team runs. The Go daemon already fixes one language boundary; the shell decides whether there's a third.

2 languages
Go (daemon) + TypeScript (renderer + supervisor)
Electron

The supervisor — spawn/adopt the daemon, health-poll, hold the token, POST /shutdown — is idiomatic Node/TS in the Electron main. The team already owns both languages.

3 languages
Go (daemon) + TypeScript (renderer) + Rust (supervisor)
Tauri

Even a "thin" Rust supervisor is a third toolchain, build, and hiring surface. The Go daemon already minimizes Tauri's Rust tax — but it does not remove the third language.

Why this is the crux. The original constraint was "stay TS." With heavy work in Go, the realistic floor is two languages (Go + TS). Electron holds that floor. Tauri raises it to three. Every other axis is either a tie (engines, security) or noise (bundle, RAM) for this app — so the language axis is what's left standing, and it points at Electron. This is also exactly the choice two fresh same-domain teams made (see Evidence).

The engine matrix — why it stopped mattering

Electron ships one pinned Chromium everywhere. Tauri uses each OS's WebView. This was the original reason Electron led. Dropping Linux removed the only weak engine, which turned a Tauri risk into a wash — not into a Tauri advantage.

Evidence gathered

Five inputs. The two that read actual code — the Codex repo reviews and the same-domain teardown — carried the most weight.

Why the shell barely matters here

The heavy work is already external. The shell renders web UI and supervises a process. Identical web client in either shell — only the supervisor language and the sidecar bundling differ.

Repo-grounded landmines

These outrank the framework choice and are shell-agnostic. Address them regardless of Electron vs Tauri.

Unauthenticated /mux WebSocket + no per-run token terminal_mux.go:35

Same-origin checks are disabled and there is no token gate, while the protocol exposes open/data/resize/close PTY frames (protocol.go:21) and terminal opens are keyed only by caller-supplied IDs (manager.go:121). The token does not exist yet: running.json persists only PID/port/start time (runfile.go:19) and config has no token env (config.go:69). A malicious local page can drive a shell. Priority #1, shell-agnostic.

No CORS for direct renderer → daemon api.go:64

The README states the daemon has no auth, CORS, or TLS (README.md:94) and /api/v1 mounts without CORS middleware. A renderer hitting 127.0.0.1 directly needs a narrow CORS allow for the app origin plus the token — or the supervisor must proxy (which defeats the thin-shell design). Pair this with the token work.

Windows terminal streaming unsupported pty_windows.go:10

The Go daemon returns "not supported yet" on Windows. Windows is a committed target. Needs ConPTY in the daemon — shell choice does not fix it.

Renderer is ~0% built frontend/src/main.ts:697

Today it is a single-file mock UI in a data: URL with "Terminal connection pending backend mux" — no React, no xterm, no supervisor. This is a build, not a shell swap. It is also why the shell choice was cheap to make either way.

Build plan

Priority order. Security and the Windows backend gap come before any renderer work. None of steps 1–2 depend on the shell.

  1. Token-gate /mux + /api/v1, add loopback CORSsecurity blocker

    Generate a per-run token at daemon start, persist in running.json, require it on route handlers. Add a narrow CORS allow for the app origin. Keep loopback-only. Touches config.go, runfile.go, terminal_mux.go, api.go.

  2. ConPTY in the Go daemon

    Implement Windows PTY (pty_windows.go) so terminals work on the platform you ship. Shell-agnostic.

  3. Electron shell + TS supervisor

    Electron main (TypeScript): bundle the ao binary via extraResources, resolve the packaged path, chmod +x on unix, spawn it, poll /readyz, read running.json, hold the token, POST /shutdown on quit. Sign + notarize the nested binary in the afterSign hook (the one Tauri-would-have-automated step — wire it once).

  4. Renderer + terminal bypass

    React 19 + Vite + Tailwind/shadcn + TanStack Query and Router (don't skip the router — Conductor's #1 perf fix) + Zustand + xterm.js (canvas default, WebGL gated) + lucide. Daemon I/O via fetch + raw WebSocket to 127.0.0.1 using generated schema.ts types. Keep this layer shell-agnostic so a future re-host is weeks, not months.

  5. One-engine QA + electron-updater

    You ship one Chromium on both OS, so terminal QA is single-engine. Wire electron-updater for signed auto-updates (mac notarized, win signed).

// electron-builder.yml — bundle + sign the Go daemon as a resource
extraResources:
  - from: "../backend/dist/ao-${arch}"
    to: "ao"
afterSign: "scripts/notarize.cjs"   // sign the nested ao binary too

// main.ts — resolve, spawn, supervise (TypeScript, team-owned)
const bin = app.isPackaged
  ? path.join(process.resourcesPath, "ao")
  : path.join(__dirname, "../backend/dist/ao");
if (process.platform !== "win32") fs.chmodSync(bin, 0o755);
const daemon = spawn(bin, ["--port", "0"]);  // reads running.json for port + token
Loose end. The accepted stack ADR predates this record. Rewrite it to Electron + mac/win + the thin-TS-supervisor-over-Go-daemon shape, and delete the Linux/AppImage targets, so the doc matches reality.
